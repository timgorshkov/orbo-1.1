## 1. Принятые бизнес-решения

### 1.1. Юридическая модель

- ООО "ОРБО" работает на **УСН 6%**.
- Модель: **агентский договор** — ОРБО является посредником, а не продавцом билетов.
- Доход ОРБО = **сервисный сбор с участников** + **агентское вознаграждение с организаторов-юрлиц** + **лицензионная подписка**.
- "Тело билета" — деньги организатора, проходящие через ОРБО транзитом (не включаются в доход ОРБО по ст. 251 НК РФ).

### 1.2. Ограничения текущего этапа (MVP)

| Ограничение | Описание |
|---|---|
| **Организаторы** | Физлица и юрлица/ИП **на УСН без НДС**. Организаторы с НДС ≠ 0 — заявка принимается, но выводится предупреждение: "В данный момент мы не работаем с организаторами-плательщиками НДС, т.к. не можем обеспечить полноценный документооборот" |
| **Участники** | Только **физлица**. Оплата от юрлиц не принимается |
| **Способ расчёта в чеке** | "Полный расчёт" при поступлении оплаты (кассовый метод) |
| **ОФД** | Провайдер ОФД пока не выбран. Реализуем формирование чеков по всем требованиям и ведение статусов, но **без отправки в ОФД** |
| **Возвраты** | Реализуются **в последнюю очередь**, после тестирования основного потока на реальных пользователях |

### 1.3. Ставки сборов и комиссий

| Тип организатора | Сервисный сбор (с участника) | Агентское вознаграждение (с организатора) | Итого доход ОРБО |
|---|---|---|---|
| **Физлицо** | **10%** от цены билета | **0%** | 10% |
| **Юрлицо / ИП** | **5%** от цены билета | **5%** от цены билета | 10% |

**Примечания для UI:**
- Для физлиц: мелким шрифтом — *"Расходы на платёжные комиссии и фискализацию берёт на себя Орбо"*
- Для юрлиц/ИП: *"Расходы на платёжные комиссии и фискальные чеки входят в агентское вознаграждение"*

### 1.4. Модель ценообразования

**Организатор задаёт "Стоимость билета для участника"** — это итоговая сумма, которую платит участник. Система автоматически выделяет из неё номинальную стоимость билета и сервисный сбор.

> При вводе цены в форме создания события, под полем автоматически показывается подсказка:
> *"Номинальная стоимость билета: XXX ₽"* (вычисленная из введённой цены)

**Формулы:**
- `total_price` = то, что вводит организатор (стоимость для участника)
- `ticket_price` (номинальная) = `total_price / (1 + service_fee_rate)`
- `service_fee_amount` = `total_price - ticket_price`
- Округление: `ticket_price` округляется вниз до копеек (`Math.floor(...*100)/100`), `service_fee_amount = total_price - ticket_price` — так сумма всегда сходится точно

### 1.5. Расчётный пример

**Организатор задаёт стоимость для участника = 1000 ₽**

| | Физлицо-организатор (сб. 10%) | Юрлицо-организатор (сб. 5%) |
|---|---|---|
| Участник платит | **1000 ₽** | **1000 ₽** |
| Номинальная стоимость билета | 909.09 ₽ | 952.38 ₽ |
| Сервисный сбор ОРБО | 90.91 ₽ | 47.62 ₽ |
| Поступает на баланс организатора | **909.09 ₽** | **952.38 ₽** |
| Агентское вознаграждение (5%) | **0 ₽** | **-47.62 ₽** |
| Итого доступно организатору | **909.09 ₽** | **904.76 ₽** |
| Итого доход ОРБО | **90.91 ₽** | **95.24 ₽** |

> **Важно:** Округление выполняется так, чтобы `ticket_price + service_fee_amount == total_price` всегда с точностью до копейки. `ticket_price = floor(total_price / (1 + rate) * 100) / 100`, `service_fee = total_price - ticket_price`.

### 1.6. Фискальные чеки (54-ФЗ)

Каждый чек содержит **две позиции**:

1. **"Билет / Услуга участия"** — с агентскими тегами (тег 1224 «данные поставщика», тег 1226 «ИНН поставщика»), чтобы выручка относилась к организатору, а не к ОРБО
2. **"Сервисный сбор Orbo"** — собственная реализация ОРБО, без агентских тегов (чистый доход ОРБО)

### 1.7. Отображение в разделе Финансы организатора

- Поступившие деньги за вычетом сервисного сбора отображаются на балансе
- Сервисный сбор **нигде не отображается** организатору (это деньги ОРБО)
- Для юрлиц: удерживаемое агентское вознаграждение **отображается** в транзакциях

### 1.8. Отчёты и акты

- Ежемесячное формирование **отчётов-актов агента** с реестром заказов, возвратов, сумм к перечислению и удержанного вознаграждения

### 1.9. Управление оплатами в событии

- Если для события включён приём платежей через ОРБО → **ручное редактирование** статуса оплаты и способа оплаты **блокируется**
- Можно редактировать стоимость билета, **пока он не оплачен**
- Если билет оплачен → доступна кнопка "Вернуть оплату" (реализуется в последнюю очередь)

### 2.3. Ключевое расхождение текущей и целевой модели

**Сейчас:** Участник платит X₽ → на баланс организатора зачисляется X₽ → из баланса удерживается комиссия (X × 5%)

**Целевая модель:** Организатор задаёт «стоимость для участника» = X₽ → система выделяет номинальную цену билета = X/(1+rate) и сервисный сбор = X - номинальная → участник платит X₽ → на баланс организатора зачисляется номинальная цена → сервисный сбор уходит ОРБО → для юрлиц из баланса дополнительно удерживается агентское вознаграждение

## 3. Целевая модель

### 3.1. Структура платежа

> Организатор задаёт «стоимость для участника» = 1000 ₽. Система разделяет:

```
┌─────────────────────────────────────────────────────┐
│     Организатор задаёт: стоимость для участника     │
│                      1000 ₽                          │
│   Система рассчитывает: ticket_price + service_fee   │
│        (физ: 909.09 + 90.91 = 1000 ₽)               │
│        (юр:  952.38 + 47.62 = 1000 ₽)               │
└────────────────────┬────────────────────────────────┘
                     │
        ┌────────────┴────────────────┐
        │                             │
   ┌────▼──────────┐          ┌──────▼──────┐
   │ Номинал билета│          │ Серв. сбор  │
   │ 909.09 ₽      │          │ 90.91 ₽     │
   │ → баланс      │          │ → ОРБО      │
   │ орг-ра        │          │ (не в леджер│
   └────┬──────────┘          │  орг-ра)    │
        │                     └─────────────┘
        │ (только для юрлиц)
   ┌────▼──────────────┐
   │ Агент. вознагр.   │
   │ -47.62 ₽ (5%)     │
   │ → удержание из    │
   │   баланса орг-ра  │
   └───────────────────┘
```

### Фаза 1: Модель данных — разделение сборов + фискальные чеки

#### 1.1. Миграция 275: Разделение сборов в org_accounts

```sql
-- Новые поля вместо единой commission_rate
ALTER TABLE org_accounts
  ADD COLUMN service_fee_rate NUMERIC(5,4) NOT NULL DEFAULT 0.1000,        -- 10% по умолчанию (физлицо)
  ADD COLUMN agent_commission_rate NUMERIC(5,4) NOT NULL DEFAULT 0.0000;   -- 0% по умолчанию

-- Обратная совместимость: commission_rate остаётся, но помечается как deprecated
COMMENT ON COLUMN org_accounts.commission_rate IS 'DEPRECATED. Использовать service_fee_rate + agent_commission_rate';
COMMENT ON COLUMN org_accounts.service_fee_rate IS 'Сервисный сбор (с участника). 0.1000 = 10% для физлиц, 0.0500 = 5% для юрлиц';
COMMENT ON COLUMN org_accounts.agent_commission_rate IS 'Агентское вознаграждение (с организатора). 0.0000 для физлиц, 0.0500 для юрлиц';
```

### Фаза 2: Платёжный поток — сервисный сбор в расчётах

#### 2.1. Сервис расчёта сборов (НОВЫЙ)

**Файл:** `lib/services/feeCalculationService.ts`

```typescript
interface FeeCalculation {
  totalAmount: number;          // стоимость для участника (то, что задал организатор)
  ticketPrice: number;          // номинальная цена билета (выделена из totalAmount)
  serviceFeeRate: number;       // ставка сервисного сбора
  serviceFeeAmount: number;     // сумма сервисного сбора
  agentCommissionRate: number;  // ставка агентского вознаграждения
  agentCommissionAmount: number; // сумма агентского вознаграждения
  organizerNetAmount: number;   // сумма на балансе организатора после всех удержаний
}

// Рассчитывает все суммы из стоимости для участника (totalAmount)
// Округление: ticketPrice = floor(totalAmount / (1 + rate) * 100) / 100
//             serviceFee = totalAmount - ticketPrice (гарантирует точное совпадение)
function calculateFees(totalAmount: number, counterpartyType: 'individual' | 'legal_entity'): FeeCalculation

// Получает тип контрагента из договора организации
function getOrgFeeConfig(orgId: string): Promise<{ counterpartyType, serviceFeeRate, agentCommissionRate }>
```

#### 2.2. Доработка paymentService.ts

- `initiatePayment()`:
  - Принимает `totalAmount` (стоимость для участника, как задал организатор)
  - Вызывает `calculateFees()` для выделения номинальной цены и сервисного сбора
  - Записывает в `payment_sessions`: `ticket_price`, `service_fee_amount`, `service_fee_rate`, `amount` (= totalAmount)
  - В шлюз отправляет `amount` (полную сумму = totalAmount)

- `markSessionSucceeded()`:
  - Вызывает обновлённый `record_incoming_payment()` с разделёнными суммами
  - Создаёт запись в `platform_income`
  - Создаёт запись в `fiscal_receipts` (статус `created`)

#### 2.3. Доработка orgAccountService.ts

- Обновить интерфейс `OrgAccount`: добавить `service_fee_rate`, `agent_commission_rate`
- `getOrCreateOrgAccount()`: при создании определять ставки по типу контрагента
- `recordIncomingPayment()`: использовать новую RPC с разделением сборов

#### 2.4. Доработка регистрации на событие

> Возвращение опции "Банковский перевод", но с ключевым отличием: деньги идут на расчётный счёт ООО "ОРБО" (не напрямую организатору), ОРБО выступает платёжным агентом, выбивает фискальный чек через OrangeData (облачная ККТ).

#### 9.1. Почему скрыт банковский перевод (текущее состояние)

Прямой перевод на счёт организатора был отключён по 4 причинам:
1. **Организатор не предупреждён** — он ожидает, что ОРБО берёт приём платежей на себя
2. **ОРБО лишается сборов и комиссий** — деньги идут мимо
3. **Риск потери платежа** — банковский перевод не мгновенный, администратор может не заметить
4. **Отсутствие фискализации** — организатор обязан выбить чек при приёме денег от физлиц, мы об этом не предупреждаем

#### 9.2. Целевая модель (банковский перевод v2)

**Схема:**
```
Участник → оплата по QR / реквизитам → р/с ООО "ОРБО"
  → OrangeData (ККТ) → ОФД → ФНС + чек участнику
  → ОРБО записывает в леджер, удерживает сборы/комиссии
  → Организатор получает вывод за вычетом сборов (как при оплате картой)
```

**Реквизиты в UI**: реквизиты **ООО "ОРБО"** (не организатора), QR-код для быстрой оплаты по СБП / банковскому приложению.

#### 9.3. Фискализация через OrangeData

**Провайдер:** OrangeData (orangedata.ru) — облачная ККТ, API `https://api.orangedata.ru:12003/api/v2/`.

**Организационные предпосылки:**
- Договор с OrangeData (аренда ККТ от 2000 руб/мес, без лимита чеков)
- Договор с ОФД (выбирается в ЛК OrangeData, 13 партнёров)
- Покупка ФН (фискальный накопитель: 12100 руб/15 мес или 18000 руб/36 мес)
- Генерация ключей (mTLS + X-Signature) через утилиту Nebula.KeysGenerator

**API чека (приход):**
```json
POST /api/v2/documents/
{
  "id": "pay_evt_{registrationId}",
  "inn": "9701327025",
  "content": {
    "ffdVersion": 4,
    "type": 1,
    "customerContact": "email_участника",
    "positions": [
      {
        "text": "Участие: {название события}",
        "price": 909.09,
        "quantity": 1,
        "tax": 6,
        "paymentMethodType": 4,
        "paymentSubjectType": 4,
        "agentType": 64,
        "supplierINN": "{ИНН организатора}",
        "supplierInfo": { "name": "{Название}", "phoneNumbers": ["{Телефон}"] }
      },
      {
        "text": "Сервисный сбор Orbo",
        "price": 90.91,
        "quantity": 1,
        "tax": 6,
        "paymentMethodType": 4,
        "paymentSubjectType": 4
      }
    ],
    "checkClose": {
      "payments": [{ "type": 2, "amount": 1000.00 }],
      "taxationSystem": 1
    }
  }
}
```

**Возврат (income_return):** тот же эндпоинт, `content.type = 2`.

**Аутентификация:** mTLS (клиентский сертификат) + X-Signature (SHA256-RSA подпись тела запроса).

**Официальный Node.js SDK:** `node-orangedata` (https://github.com/orangedata-official/node-orangedata).

#### 9.4. Сверка банковских переводов (reconciliation)

**Админ-интерфейс в суперадмин-панели:**
- Страница загрузки банковской выписки (CSV/1C формат) или ручного ввода
- Парсинг назначений платежа — поиск `ORB-XXXX-XXXX` payment reference
- Автоматическое сопоставление с pending payment_sessions
- Кнопка "Подтвердить" для каждого найденного платежа → вызов `reconcileBankTransfer()`
- Batch-подтверждение для множественных совпадений

**Существующий код:** `reconcileBankTransfer()` в `paymentService.ts` уже реализована — ищет сессию по `payment_reference` и подтверждает.

#### 9.5. QR-код для быстрой оплаты

- Генерация QR-кода с зашитыми реквизитами ОРБО + назначением платежа `ORB-XXXX-XXXX`
- Формат: стандартный банковский QR по ГОСТ (поддерживается всеми банковскими приложениями)
- Показывается на странице оплаты рядом с реквизитами
- Пользователь сканирует QR → открывается банковское приложение с заполненными полями

#### 9.6. Интеграция в payment flow

1. `fiscalReceiptService.ts` → добавить `sendToOrangeData()` — маппинг `FiscalReceipt` → OrangeData API
2. `paymentService.ts` → `markSessionSucceeded()` вызывает `createPaymentReceipt()` → `sendToOrangeData()`
3. `paymentService.ts` → `processRefund()` вызывает `createRefundReceipt()` → `sendToOrangeData(type=2)`
4. Крон `check-receipt-status` → поллинг `GET /api/v2/documents/{inn}/status/{id}`
5. Суперадмин: страница reconciliation для банковских переводов

## Приложение А: Справочник юридических требований

### Агентские теги в чеке (ФФД 1.2)

| Тег | Название | Значение |
|---|---|---|
| 1057 | Признак агента | Агент |
| 1222 | Признак агента по предмету расчёта | Агент |
| 1224 | Данные поставщика | Наименование/ФИО организатора, контактный телефон |
| 1226 | ИНН поставщика | ИНН организатора |
| 1214 | Способ расчёта | "Полный расчёт" (для MVP) |
| 1212 | Признак предмета расчёта | "Услуга" |

### Реквизиты ОРБО для чеков

```
Наименование: ООО «ОРБО»
ИНН: 9701327025
КПП: 770101001
ОГРН: 1267700119037
Система налогообложения: УСН (доходы)
```

### Критические правила

1. **Без ИНН поставщика (тег 1226) в чеке** → ФНС квалифицирует ОРБО как поставщика → вся выручка = доход ОРБО
2. **Мониторинг порога УСН-НДС**: 20 млн ₽ в 2026, 15 млн в 2027, 10 млн в 2028 — считать только по доходам ОРБО (сервисный сбор + агентское вознаграждение + подписки)
3. **Выплаты физлицам**: не удерживать ничего из денег физлица → сервисный сбор берётся с участника отдельно

- [ ] Все оплаты через ОРБО формируют запись `fiscal_receipts` с корректными позициями
- [ ] В позиции "Билет" всегда заполнены `supplier_name` и `supplier_inn` (из counterparties)
- [ ] Нельзя создать платёжную сессию без ИНН организатора в контрагентах
- [ ] Сервисный сбор отображается участнику ДО оплаты
- [ ] Ручное редактирование payment_status заблокировано при ОРБО-платежах
- [ ] Ставки сборов автоматически устанавливаются по типу контрагента

- [ ] Леджер организатора НЕ содержит записей о сервисном сборе
- [ ] Для юрлиц в леджере видно агентское вознаграждение
- [ ] Суперадмин может переопределить ставки для конкретной организации
- [ ] Ежемесячные отчёты агента генерируются автоматически
